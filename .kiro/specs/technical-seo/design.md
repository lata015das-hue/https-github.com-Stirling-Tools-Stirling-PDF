# Technical SEO — Design

## 1. Architecture

```
       data/site-config.json (or env SITE_URL)
                  │
                  ▼
  ┌────────────────────────────────────┐
  │  domain resolver (single source)   │
  └────────────────────────────────────┘
       ┌──────────┬───────────┬──────────────┬──────────────────┐
       ▼          ▼           ▼              ▼                  ▼
   sitemap.xml  robots.txt  canonical    OG tags         JSON-LD slot
                            in <head>    in <head>      (Schema Gen hook)
                                                         │
                                                ▼
                                       <script type="application/ld+json">
```

The principle: **one place to change the domain**, then every emitter consumes
it. Today the domain lives in three places (sitemap.xml, robots.txt, the
suggestedPage paths in keyword-strategy.json) — that's the bug task 1 fixes.

## 2. Components

### 2.1 Site-config resolver

A single function `getSiteUrl()` that returns the production hostname:

1. Reads `SITE_URL` environment variable if set.
2. Else reads `siteUrl` from `data/site-config.json`.
3. Else falls back to `https://example.com` and emits a `::warning::` so CI
   surfaces the placeholder.

Used by every script that emits a URL.

### 2.2 Sitemap (already partially exists)

`sitemap.xml` is regenerated by the `sitemap-updater` hook on file create/delete
under `src/routes/**/*.tsx` and `content/**/*.mdx`. The hook needs three
extensions:

- Read host from `getSiteUrl()` instead of inferring from existing `<loc>`.
- Emit `<xhtml:link rel="alternate" hreflang="…">` per URL group when an `ar/`
  counterpart exists.
- Emit `x-default` pointing at the English variant.

### 2.3 Robots

`robots.txt` is one-line generated; same `getSiteUrl()` consumer. No User-agent
rules beyond `User-agent: * / Allow: /`.

### 2.4 Canonical & lang in `<head>`

Today only `index.html` has `<html lang="en" dir="ltr">`. Once
`tool-pages-seo` lands, every static page sets its own `lang`/`dir` via the
layout component (`_ToolStub.tsx`).

For SPA URLs (`index.html#/...`), canonicalization is handled differently:
the SPA emits `<link rel="canonical">` dynamically based on the current route,
or — once static pages exist — the SPA URL becomes a redirect target and the
static URL is the canonical.

### 2.5 hreflang cluster

Spec: each public URL emits in `<head>`:

```html
<link rel="alternate" hreflang="en" href="https://<host>/<en-path>">
<link rel="alternate" hreflang="ar" href="https://<host>/ar/<en-path>">
<link rel="alternate" hreflang="x-default" href="https://<host>/<en-path>">
```

The cluster is computed by walking the file tree:

- `src/routes/tools/<slug>.tsx` → en variant
- `src/routes/tools/<lang>/<slug>.tsx` → `<lang>` variant
- `content/blog/<slug>.mdx` → en blog variant
- `content/blog/<lang>/<slug>.mdx` → `<lang>` blog variant

The `audit_i18n.py` script already enumerates these pairs; it just doesn't yet
emit the hreflang HTML.

### 2.6 JSON-LD via Schema Generator hook

The `schema-generator` hook is manually triggered today, runs against an open
HTML file, picks a schema type (FAQPage > HowTo > SoftwareApplication > Product
> Article), and inserts a `<script type="application/ld+json"
data-generated="schema-generator">` block before `</head>`.

The technical-SEO commitment is: every shipped page has run through this hook
once, with the editor reviewing the output. No automated invocation in CI yet —
schema generation is assistive, not enforced. Enforcement (presence + JSON
validity) lives in the SEO Validator hook (step 4 of the user's plan).

## 3. The 301 / canonical question

When static pages ship, the existing SPA URL `index.html#/tool/<id>` is the
canonical bridge. Two options:

- **Option A:** Static page is canonical, SPA URL stays a working route the
  static page links into. **Tradeoff:** users who land on the SPA URL don't get
  the SEO copy.
- **Option B:** Static page is canonical *and* the SPA route emits a 302 to
  the static page on first hit. **Tradeoff:** complicates the SPA bootstrap.

This spec defaults to A (simpler), revisit if traffic data shows users landing
on `#/tool/...` URLs from social media or saved bookmarks.

## 4. Tasks 1–4 (the user's plan)

The task list (see `tasks.md`) implements the design above in this order:

1. Site-config resolver + sitemap/robots consume it (largest immediate win).
2. Canonical + lang on `index.html` is correct and forward-compatible.
3. hreflang scaffolding so the cluster is *ready* the moment Arabic routes ship.
4. 301 strategy doc (Option A vs. Option B) + lock the choice.
